RCCP Mirror Networking Integration Guide

This guide covers everything you need to set up, configure, and test multiplayer with Realistic Car Controller Pro using Mirror Networking. By the end of this guide, you will have a working multiplayer car game where players can host, join, and drive vehicles together over a network.

Prerequisites

Before starting, make sure you have the following:

RequirementDetails
Unity Version6000.0.49f1 or later
RCCPRealistic Car Controller Pro imported and working
MirrorMirror Networking package imported from the Asset Store or GitHub
TransportKCP Transport (included with Mirror, recommended for games)

Mirror must be imported before the RCCP Mirror integration package. If Mirror is not present in your project, the integration scripts will not compile.

Installation

Step 1 — Import Mirror

Import Mirror Networking into your Unity project from the Asset Store or the official Mirror repository. Confirm it compiles without errors before proceeding.

Step 2 — Import the RCCP Mirror Integration Package

Navigate to Assets/Realistic Car Controller Pro/Addons/Installers/ in your Project window. Double-click RCCP_MirrorIntegration.unitypackage and import all files.

After import, the editor will automatically:

  1. Add the RCCP_MIRROR scripting define symbol to your project
  2. Display a setup dialog if your project uses URP or HDRP, offering to convert materials

The integration files are installed to Assets/Realistic Car Controller Pro/Addons/Installed/Mirror/.

Step 3 — Verify

After import, check the Console for any compilation errors. If everything is correct, you should see no errors related to Mirror or RCCPMIRROR. A demo scene is included at Addons/Installed/Mirror/Scenes/RCCPSceneBlankMirror.unity that you can open to verify the setup.

Setting Up a Vehicle for Mirror

To make any RCCP vehicle work with Mirror networking, you need two components added to the root vehicle GameObject.

Required Components

ComponentPurpose
NetworkIdentityMirror's core networking component. Identifies this object on the network.
RCCP_MirrorSyncRCCP's synchronization script. Handles replication of vehicle state across clients.

Step-by-Step

  1. Select your vehicle prefab in the Project window (or an instance in the scene).
  2. The vehicle must already have an RCCP_CarController component on it (standard RCCP setup).
  3. Add a NetworkIdentity component to the root GameObject (same object as RCCP_CarController).
  4. Add an RCCP_MirrorSync component to the same root GameObject.
  5. Save the prefab.

That is all that is required. RCCP_MirrorSync automatically finds and caches all RCCP subsystems (Engine, Gearbox, Differential, Wheels, Lights, etc.) on Awake.

Reference Prefab

A pre-configured vehicle prefab is included at: 


Assets/Realistic Car Controller Pro/Addons/Installed/Mirror/Prefabs/

You can duplicate this prefab and swap the 3D model to create additional networked vehicles, or use it as a reference for how the components should be arranged.

Creating a Multiplayer Scene

This section walks you through creating a Mirror multiplayer scene from scratch.

Step 1 — Create a New Scene

Create a new Unity scene. Add your environment, ground plane, lighting, and any other static objects you need.

Step 2 — Add RCCP Scene Essentials

Add the standard RCCP scene components. You can do this via the RCCP menu:

  • RCCP_SceneManager — Required for RCCP to function. Manages the active player vehicle and camera.
  • RCCP_Camera — The player camera that follows the active vehicle.
  • RCCP_SkidmarksManager — Optional but recommended for tire mark effects.

Step 3 — Add a NetworkManager

Create an empty GameObject and name it NetworkManager. Add the following components:

ComponentConfiguration
NetworkManagerSet Player Prefab to your Mirror-ready vehicle prefab. Set Player Spawn Method to Round Robin or Random.
KCP TransportDefault settings work well. Port 7777 is standard.
NetworkManagerHUDOptional. Provides a simple Host/Join UI panel for testing. Remove for production.

The Player Prefab field is the most important setting. This is the vehicle that Mirror will automatically spawn for each player when they connect. It must have NetworkIdentity and RCCP_MirrorSync components.

Step 4 — Add Spawn Points

Create empty GameObjects at the positions where you want vehicles to appear. Add the NetworkStartPosition component to each one. Place them with enough spacing so vehicles do not overlap when multiple players connect.

Mirror's NetworkManager automatically finds all NetworkStartPosition objects in the scene and distributes players across them based on the Player Spawn Method setting:

  • Round Robin — Cycles through spawn points in order (first player gets point 1, second gets point 2, etc.)
  • Random — Picks a random spawn point for each player

Step 5 — Register the Scene in Build Settings

Open File > Build Settings and add your scene to the Scenes In Build list. This is required for Mirror to load scenes correctly during networked play.

Step 6 — Register Vehicle Prefabs

All vehicle prefabs that may appear in the scene must be known to Mirror. There are two ways to handle this:

Option A — Inspector (Recommended): Drag your vehicle prefab into the NetworkManager's Registered Spawnable Prefabs list in the Inspector.

Option B — Automatic: RCCPMirrorSync has a static method EnsureSpawnInfrastructure() that automatically registers all vehicles from the RCCP vehicle list with the NetworkManager at runtime. This is called automatically by RCCPMirrorManager.OnStartServer(). If you are using the default NetworkManager instead of RCCP_MirrorManager, call this method manually from your own startup script.

Using the Lobby System

RCCP includes RCCP_MirrorManager, a full lobby system with LAN server discovery, server browsing, nicknames, and chat. Use this instead of the default NetworkManager when you want a complete multiplayer menu experience.

Setup

  1. Replace the default NetworkManager component with RCCP_MirrorManager on your NetworkManager GameObject.
  2. Add a NetworkDiscovery component to the same GameObject for LAN server browsing.
  3. Wire up the UI references in the Inspector (nick panel, room browser, chat panel, etc.).
  4. Set the Gameplay Scene Name field if you want the host to automatically load a different scene after creating a room.

How It Works

RCCP_MirrorManager inherits from Mirror's NetworkManager and adds:

  • Nickname entry — Players enter a name before connecting
  • LAN server discovery — Automatically finds servers on the local network
  • Server browser UI — Lists discovered servers with join buttons
  • Host/Join flow — Create a room (host) or join a discovered server (client)
  • Auto playerPrefab — If no playerPrefab is assigned in the Inspector, it automatically uses the first vehicle from the RCCP vehicle list

Connection Flow

Hosting:

  1. Player enters a nickname and clicks Connect
  2. LAN discovery starts, showing available servers
  3. Player clicks "Create Room" to host
  4. Server starts, discovery begins advertising
  5. If a gameplay scene is configured, the server loads it
  6. Host's vehicle is auto-spawned via playerPrefab

Joining:

  1. Player enters a nickname and clicks Connect
  2. LAN discovery finds available servers
  3. Player selects a server from the list and clicks Join
  4. Client connects to the selected server
  5. Client's vehicle is auto-spawned via playerPrefab

Using the Simple Demo Setup

For quick testing without a lobby system, you can use Mirror's built-in NetworkManagerHUD component. This draws a simple IMGUI panel with Host, Client, and Server buttons.

  1. Add NetworkManager and KcpTransport to a GameObject
  2. Add NetworkManagerHUD to the same GameObject
  3. Set the Player Prefab to your Mirror-ready vehicle prefab
  4. Add NetworkStartPosition objects for spawn points
  5. Enter Play mode, click Host, and you are driving

For a second player, build the project and run the build. Click "Client" and enter the host's IP address (or leave as localhost for same-machine testing).

How the System Works

Auto-Spawn on Connect

Mirror's NetworkManager has a built-in mechanism for spawning player objects. When a client connects, the server automatically instantiates the playerPrefab and calls AddPlayerForConnection, which:

  1. Spawns the vehicle on the server
  2. Replicates it to all clients
  3. Sets isLocalPlayer = true on the owning client immediately
  4. Grants client authority over the object

This means the vehicle appears and is controllable from the very first frame — no manual spawn button needed.

Vehicle State Synchronization (RCCP_MirrorSync)

RCCP_MirrorSync is the core networking component. It runs on every networked vehicle and handles two roles depending on ownership.

On the owner's machine (IsMine = true):

  • Normal RCCP input and physics run locally
  • 30 times per second (configurable), the full vehicle state is packed and sent to the server via a Mirror Command
  • State includes: throttle, brake, steering, handbrake, boost, clutch, engine RPM, gear state, differential outputs, wheel RPMs, light states, position, rotation, velocity, and angular velocity

On remote machines (IsMine = false):

  • Local RCCP input is disabled via the override system
  • Received network state is applied to all RCCP subsystems (inputs, engine, gearbox, clutch, differentials, lights)
  • Position and rotation are interpolated with lag compensation and optional extrapolation
  • Wheel RPM correction components are added to keep wheel visuals accurate

Ownership Detection (IsMine Property)

The IsMine property on RCCP_MirrorSync determines whether the local client owns a vehicle. It checks in order:

  1. isLocalPlayer — Set by Mirror when the vehicle is the client's player object (most reliable)
  2. isOwned — Mirror's general ownership flag
  3. Host-mode fallback — Server-side check for host-owned vehicles

If IsMine is true, the vehicle runs local physics and sends state. If false, the vehicle receives and applies remote state.

Network Data Flow 


Owner Client                    Server                     Remote Clients
     |                            |                              |
     |-- CmdSendVehicleState ---->|                              |
     |                            |-- RpcReceiveVehicleState --->|
     |                            |                              |
     |-- TransformFallback ------>|                              |
     |                            |-- RpcTransformFallback ----->|

The owner sends vehicle state via a Command (client to server). The server validates it and relays to all other clients via ClientRpc. A parallel transform fallback channel handles edge cases in host mode.

Lag Compensation and Interpolation

Remote vehicles do not run physics locally. Instead, RCCP_MirrorSync uses a multi-layered approach to keep them visually smooth:

  1. Lag compensation — When a network update arrives, the position is projected forward by the estimated network lag to compensate for transmission delay
  2. Interpolation — The vehicle smoothly moves from its current position toward the target using configurable lerp speeds
  3. Extrapolation — Between network updates, the vehicle continues moving along its last known velocity to prevent stuttering
  4. Teleportation — If the distance between current and target positions exceeds a threshold (default 4m), the vehicle teleports instantly to avoid rubberbanding

Wheel RPM Correction

Remote vehicle wheels need special handling because their physics simulation runs independently on each client. RCCP_WheelRPMCorrectionMirror is automatically added to each wheel on remote vehicles. It applies corrective motor torque to make wheel rotation match the owner's actual RPM values, keeping wheel spin visuals accurate across the network.

Configuration and Tuning

RCCP_MirrorSync Inspector Settings

SettingDefaultDescription
Teleport Distance Threshold4Meters. If a remote vehicle is farther than this from its target, it teleports instead of interpolating. Lower values make corrections snappier but more visually jarring.
Lag Compensation Time0.05Seconds. Additional forward prediction applied to compensate for network delay. Increase for high-latency connections.
Position Interpolation Speed10Multiplier for position lerp. Higher values make remote vehicles snap to targets faster.
Rotation Interpolation Speed10Multiplier for rotation slerp. Higher values reduce rotational lag.
Use ExtrapolationtrueWhen enabled, remote vehicles continue moving along their velocity between updates. Disable if you see vehicles overshooting.
Max Extrapolation Time0.1Seconds. Maximum time the system will predict ahead. Prevents runaway extrapolation during packet loss.
Use Velocity SmoothingtrueSmoothly transitions between velocity values instead of snapping.
Velocity Dampening0.8Damping factor for velocity interpolation. Lower values are smoother but less responsive.
Send Rate30Updates per second sent from owner to server. Higher values improve accuracy but use more bandwidth. 20-30 is recommended for most games.

Bandwidth Estimation

Each vehicle sends approximately 600 bytes per update. At the default 30 Hz send rate:

To reduce bandwidth for larger player counts, lower the Send Rate to 20 or even 15. The interpolation system smooths out the lower update frequency.

Building and Testing

Testing on a Single Machine

  1. Open your Mirror scene and enter Play mode — click Host
  2. Build the project (File > Build and Run)
  3. In the built application, click Client and connect to localhost
  4. Both instances should show two vehicles driving independently

Testing on a Local Network

  1. Build the project and copy the build to a second machine on the same network
  2. On machine A, run the game and click Host
  3. On machine B, run the game and click Client, entering machine A's local IP address
  4. If using RCCP_MirrorManager, the LAN discovery should find the host automatically

Testing with ParrelSync (Recommended)

ParrelSync is a free Unity editor extension that creates linked clones of your project, allowing you to test multiplayer entirely within the Unity Editor without building.

  1. Import ParrelSync from the Asset Store or GitHub
  2. Open ParrelSync > Clones Manager and create a clone
  3. Open the clone project in a second Unity Editor instance
  4. Run the main project as Host and the clone as Client

Common Build Checklist

  • All multiplayer scenes are added to Build Settings
  • Vehicle prefabs are registered with NetworkManager (either via Inspector or EnsureSpawnInfrastructure)
  • NetworkStartPosition objects are placed in the scene
  • KCP Transport port (default 7777) is not blocked by firewall

FAQ and Troubleshooting

The vehicle spawns but I cannot control it

This usually means IsMine is returning false. Check that:

  • The vehicle prefab has both NetworkIdentity and RCCP_MirrorSync components
  • The prefab is assigned as the Player Prefab on the NetworkManager
  • You are connecting as a client (not just starting the server without a client)

Remote vehicles are jittery or stuttering

Increase Position Interpolation Speed and Rotation Interpolation Speed on RCCP_MirrorSync. If the vehicles are teleporting frequently, increase Teleport Distance Threshold. For high-latency connections, increase Lag Compensation Time slightly.

Remote vehicle wheels are not spinning correctly

RCCPWheelRPMCorrectionMirror should be added automatically to remote vehicles. If wheels still look wrong, check that RCCPMirrorSync is on the root vehicle object (same level as RCCP_CarController), not on a child object.

"Replacing existing prefab with assetId" warning

This warning appears when the same prefab is registered with Mirror twice — once as playerPrefab (auto-registered by Mirror) and again in the spawnPrefabs list. It is harmless. To silence it, remove the playerPrefab from the Registered Spawnable Prefabs list since Mirror registers it automatically.

"No Transport assigned to Network Manager" warning

Mirror found a KCP Transport on the same GameObject but it was not explicitly assigned to the Transport field. Drag the KCP Transport component into the NetworkManager's Transport slot in the Inspector to silence this.

All players spawn at the same position

Add NetworkStartPosition components to empty GameObjects in your scene. Place them at different positions. Set Player Spawn Method on the NetworkManager to Round Robin or Random.

Vehicles work in editor but not in builds

Make sure the scene is added to File > Build Settings. Also verify that vehicle prefabs are in a Resources folder or are referenced by a scene/component so Unity includes them in the build.

Can I switch vehicles during a session?

Vehicle switching is not currently supported in Mirror sessions. The vehicle assigned as playerPrefab is spawned automatically when a player connects, and each player uses that vehicle for the duration of the session.

How do I use different vehicles for different players?

Override OnServerAddPlayer in a custom NetworkManager subclass. In that method, you can instantiate any vehicle prefab based on player preferences and call NetworkServer.AddPlayerForConnection with the custom vehicle instead of the default playerPrefab.

What data is synchronized over the network?

RCCP_MirrorSync synchronizes: throttle, brake, steering, handbrake, boost/NOS, clutch inputs, engine RPM, engine running state, gear state, differential outputs, wheel RPMs, all light states (headlights, indicators), and full transform (position, rotation, velocity, angular velocity).

What is NOT synchronized?

Damage state, audio, visual effects (skidmarks, exhaust), and suspension visuals are not explicitly synchronized. These are driven locally by the synchronized inputs and physics state, so they approximate the owner's state but may differ slightly on remote clients.
